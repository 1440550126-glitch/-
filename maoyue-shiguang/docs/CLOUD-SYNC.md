# 猫约时光 · 情侣云同步说明

把 `store.js` 的本地数据接到**云开发数据库**，实现你和 TA 两台手机的**双向同步**。设计为**离线优先**：本地缓存仍是页面的同步读源，云端在后台合并 —— 没网/没开通云开发时，App 照常当本地原型用。

## 涉及文件

| 文件 | 作用 |
|------|------|
| `app.js` | 启动初始化云开发；启动/回前台拉取、退后台推送 |
| `utils/cloud.js` | 云函数调用封装；未开通时安全降级为 no-op |
| `utils/store.js` | 同步层：`write()` 自动打时间戳+防抖推送；`beginSync/syncNow/bindCouple/unbindCouple` |
| `cloudfunctions/coupleData/` | **云端权威合并方**：bind / pull / sync / unbind |

## 数据模型（云数据库集合 `couples`）

```
{
  _id,                       // 情侣关系 ID
  members: [openidA, openidB],
  inviteCode: "ABC234",      // 6 位邀请码
  sync: { data:{…}, meta:{key:时间戳}, tomb:{key:[已删id]} },
  updatedAt
}
```

- 云函数以**管理员身份**读写，并以调用者 `OPENID` 鉴权（只能操作自己所属的关系）。
- 不再需要单独的 users 集合：用 `where({ members: openid })` 反查关系。

## 哪些数据会同步

✅ **共享数据（视角无关）**：愿望清单、纪念日、时光回忆、一起养的猫、小金库流水、点餐菜单、在一起的日期（`startDate`）。

- 小金库**余额由流水汇总重算**（`recomputeVault`），避免并发存取互相覆盖丢钱。
- 列表类按 **id 并集**合并：两人各自新增都不丢；删除用**墓碑**保证能同步删除。
- 列表 id 用 `uid()`（时间戳+随机），避免同一毫秒（含两台设备各自）创建时 id 冲突丢项。
- 对象类（猫/日期）按**时间戳取新**（后写覆盖）。

✅ **视角相关数据（按作者拆分后已同步）**：心情、每日一问、纸条。

- **存绝对作者、读时翻译**：每条/每项按 `openid` 标注作者；`getMood/getDailyQuestion/getNotes` 读取时按"是不是我"翻译成 `self/partner`、`from`，所以**页面代码不用改**，也**不会身份对调**。
- 心情/每日一问用 **`authormap`** 合并：`{ date, byAuthor:{ openid: 值 } }`，同一天并集两人的条目（同作者冲突取较新），换天取较新日期整条。
- 纸条按 id 并集（带 `author`）。绑定前用本地哨兵 `'me'` 记的内容，绑定时 `migrateAuthor` 改挂到真实 `openid` 再上传。
- 未绑定/未开通云开发时，对方心情/答案用温柔**模拟**填充（保留单人体验）；绑定后只显示对方真实记录。

🚫 **不同步**：昵称 / 头像 —— 这是"我怎么称呼对方"的本地偏好，各自保留。

## 合并策略（最终一致）

客户端只负责"导出本地 / 采纳云端"，**所有合并在云函数里完成**（单一权威，避免双端逻辑漂移）：

- `pull`（首次启动）：只取云端，强制采纳。
- `sync`（改动后/回前台/退后台）：上传本地 → 云端 `mergeDocs(已存, 本地)` → 保存并返回 → 客户端采纳（仅当云端某键不比本地旧时覆盖，保护未同步的新改动）。

已用单元测试覆盖：并发新增都保留、墓碑删除生效、对象按时间戳取新、id 冲突取新、删除经墓碑往返同步。

## 绑定流程（真正让两台手机共享数据）

1. A 在「我的 → 绑定关系」选「生成我的邀请码」→ 得到 6 位码，发给 B。
2. B 选「输入 TA 的邀请码」→ 填入 → 双方进入同一段关系，开始同步（B 看到鹊桥相会过场）。
3. A 下次回到前台时自动检测到 B 已加入，`bound` 置真。

未开通云开发时，绑定退化为本地标记（与原型一致）。

## 部署步骤

1. 开发者工具开通**云开发**，创建环境，记下环境 ID。
2. 把环境 ID 填到 `app.js` 顶部 `CLOUD_ENV`（留空用默认环境）。
3. 云开发控制台 →「数据库」→ 新建集合 **`couples`**（权限可设"仅管理端可读写"，因为只经云函数访问）。
4. 右键 `cloudfunctions/coupleData` →「上传并部署：云端安装依赖」。（`secCheck` 同样部署。）

## 两台设备联调测试

1. 设备 A 生成邀请码；设备 B 输入加入。
2. A 加一条愿望/纪念日/回忆/存一笔钱 → B 回前台应看到。
3. B 也加一条 → A 回前台应看到（两边都在，不互相覆盖）。
4. A 删除某条 → B 回前台应同步删除（墓碑生效）。
5. A 记心情/答每日一问/贴纸条 → B 看到的应是"TA"的（self/partner 不对调）。
6. 断网操作后再联网回前台 → 改动应补传/补收。

## 准实时（startRealtime / stopRealtime）

前台时自动开启准实时（`app.js` onShow 开、onHide 关）：

- 优先用云数据库 **`watch()`**：对方一改，本机即时收到并 `adopt`（需给 `couples` 配读权限的安全规则，见下）。
- watch 不可用/无权限时**自动回退轮询**：每 10s 调一次极轻的 `ping`（只回 `updatedAt`），比本地已知的新才真正拉一次 `sync`。
- 因此默认部署（集合"仅管理端"）也能"对方改动约 10s 内可见"；想要"即时"，给 `couples` 配如下安全规则即可启用 watch：
  ```json
  { "read": "doc.members.includes(auth.openid)", "write": false }
  ```

## 在场状态 / 戳一戳（presence）

- `K.PRESENCE`（authormap）记录每个人的 `{ ts, battery, charging, locOn, lat, lng, pokeTs, pokeType }`。
- `reportPresence()`：`app.js` 每次回前台上报电量/在线；`status` 页在**开启位置共享**时上报经纬度（默认关闭，隐私优先）。
- `getPartnerStatus()`：读对方的 presence → 真实电量/在线/距离；对方还没上报则显示"还没上线过"；**未绑定时回退温柔模拟**（保留单人体验）。
- **戳一戳**：`pokePartner(type)` 写自己的 `pokeTs` 同步出去；对方经 watch/轮询收到后，`getIncomingPoke()` 取到新戳 → `app.afterSync` 弹"TA 戳了戳你"。App 关闭时的推送可后续接**订阅消息**。

## 已知限制（后续）

- 准实时默认是 10s 轮询；配了 watch 安全规则才是即时推送。
- 删除采用墓碑（量极小）；长期可定期清理。
- 并发编辑**同一条**记录的极端情况按时间戳取新，可能丢一侧的该条改动（新增不受影响）。
- presence 每次回前台上报一次（含电量），不做高频心跳；"在线"判定为"5 分钟内有上报"。
